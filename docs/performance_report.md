# Performance Analysis Report

This report documents benchmarking scenarios, measurement methodology, results, and analysis for the Distributed Synchronization System.

**Summary:**
- Benchmarks executed with a synthetic harness (see `benchmarks/run_benchmarks.py`).
- Scenarios: `single-node`, `distributed-3node`, `geo-distributed-3node`.
- Metrics: throughput (ops/sec), latency (ms), and scalability (aggregate throughput as cluster size grows).

**How to reproduce**

1. Install benchmarking dependencies:

```bash
pip install -r requirements.txt
```

2. Run the synthetic benchmark generator:

```bash
python benchmarks/run_benchmarks.py
```

3. Output images will be written to `docs/images/`:
- `throughput.png` — Throughput vs Concurrent Clients
- `latency.png` — Latency vs Concurrent Clients
- `scalability.png` — Aggregate Throughput vs Cluster Size

If you have a live cluster up (see `docs/deployment_guide.md`), adapt `benchmarks/load_test_scenarios.py` to execute real HTTP/JSON workloads using `locust` or `requests` and compare results.

## Benchmarking Scenarios

- Single-node: run the system with a single process on localhost; measures baseline latency and throughput without replication.
- Distributed (3-node): run a 3-node cluster with replication and Raft-based consensus for lock operations.
- Geo-distributed: same 3-node cluster but nodes are assigned simulated inter-region latencies to represent cross-region penalties.

## Results (Synthetic)

Charts generated by the synthetic harness are included in `docs/images/` after running the script. Sample observations from the synthetic results:

- Throughput: single-node shows highest raw per-node throughput at low concurrency; distributed cluster achieves comparable aggregate throughput while offering resilience and availability.
- Latency: geo-distributed scenario shows substantially higher median latency due to inter-region RTTs and consensus overhead.
- Scalability: aggregate throughput improves with node count but with diminishing returns due to coordination overhead.

## Analysis

- Throughput vs Latency trade-off: distributed systems trade per-request latency for fault tolerance and higher aggregate capacity. For workloads tolerant to slightly higher latency (queues, background tasks), distributed mode is preferred.
- Bottlenecks: consensus (Raft) introduces latency for operations that require strong consistency (e.g., lock `acquire`). Caching and local reads (cache node) can mask this cost.
- Recommendations:
  - Use local cache reads where possible to reduce latency for read-heavy workloads.
  - Partition high-volume topics across nodes using consistent hashing to balance load.
  - For geo-distributed deployments, colocate latency-sensitive clients near leader nodes or use read replicas.

## Comparison: Single-node vs Distributed

- Single-node: simpler, lower latency, lower availability, limited scalability.
- Distributed: higher availability, fault tolerance, horizontal scalability, but higher coordination latency.

## Appendix: Suggested Real Benchmarks

- Lock throughput test: issue concurrent `acquire` requests for distinct `lock_id`s and measure ops/sec and p50/p95 latency.
- Contended lock test: multiple clients contend for the same `lock_id` to observe queuing and latency amplification.
- Queue throughput: measure enqueue/dequeue rates with multiple producers/consumers.
- Cache hit ratio: measure read latency differences with warm vs cold caches.
